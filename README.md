# composer-merge-driver

A modern PHP 8.1+ git merge driver that semantically resolves conflicts in `composer.json`, `composer.lock`, and `vendor/composer/*` files — without producing invalid JSON or mangled lock files.

Unlike text-based merges, this driver understands Composer's file structure and can re-invoke `composer update` to re-resolve package versions when a conflict cannot be solved structurally.

---

## Why this exists

Git's default line-based merge produces broken results for Composer files:

| Problem | What git does | What this driver does |
|---|---|---|
| Trailing comma on last `require` entry | Leaves invalid JSON | Parses and re-serializes at the object level |
| Two branches each add a package | Produces conflict markers inside JSON | Merges package maps independently |
| Lock file version conflict | Conflict markers inside JSON | Re-runs `composer update <pkg>` in an isolated temp dir |
| `vendor/composer/*` auto-generated files | Conflict markers | Accepts "ours" — regenerated by `composer install` |

## Installation

### Option A — global (recommended for local development)

```bash
composer global require spencermalone/composer-merge-driver
```

The binary is then available at `~/.composer/vendor/bin/composer-merge-driver` (or on your `$PATH` if you have the global Composer bin dir configured).

### Option B — per-project dev dependency

```bash
composer require --dev spencermalone/composer-merge-driver
```

Use `./vendor/bin/composer-merge-driver` as the driver path.

### Option C — Docker (see below)

---

## Git configuration

### 1. Add to `.gitattributes` (commit this file)

```
composer.json     merge=composer-merge
composer.lock     merge=composer-merge
vendor/composer/* merge=composer-merge
```

### 2. Register the driver (per developer or in a team setup script)

**Local install:**
```ini
# .git/config  or  ~/.gitconfig
[merge "composer-merge"]
    name   = Composer merge driver
    driver = composer-merge-driver %O %A %B %L %P
```

**Per-project with `./vendor/bin`:**
```ini
[merge "composer-merge"]
    name   = Composer merge driver
    driver = ./vendor/bin/composer-merge-driver %O %A %B %L %P
```

## Usage modes

### Local environment

The simplest setup. The driver binary must be on `$PATH`:

```ini
[merge "composer-merge"]
    driver = composer-merge-driver %O %A %B %L %P
```

If `composer.lock` conflicts contain version disagreements, the driver automatically spawns `composer update <package>` in an isolated temp directory to re-resolve them. Requires internet access and a working `composer` binary.

To skip re-resolution (e.g. in CI or offline environments):

```ini
[merge "composer-merge"]
    driver = composer-merge-driver --no-resolve %O %A %B %L %P
```

---

### Docker — `docker exec` (static container)

Best when you already run a container that has the project mounted inside it.

```ini
[merge "composer-merge"]
    name   = Composer merge driver
    driver = docker exec my-php-container \
             /app/vendor/bin/composer-merge-driver \
             --working-dir /app \
             %O %A %B %L %P
```

The container must have the driver installed and the project mounted at the path you specify with `--working-dir`.

---

### Docker — `docker run` (on-demand container)

Spawns a fresh container for every merge operation. The project directory is mounted at the same absolute path so temp file paths (%O, %A, %B) remain valid inside the container.

```bash
# In your shell profile, define the mount root:
REPO_ROOT=$(git rev-parse --show-toplevel)
```

```ini
[merge "composer-merge"]
    name   = Composer merge driver
    driver = docker run --rm \
             -v "$HOME/.composer:/root/.composer:ro" \
             -v "$(git rev-parse --show-toplevel):/project" \
             ghcr.io/spencermalone/composer-merge-driver \
             --working-dir /project \
             %O %A %B %L %P
```

Mounting `~/.composer` (read-only) gives the container access to Composer auth credentials for private packages during re-resolution.

## Options

| Flag | Default | Description |
|---|---|---|
| `--working-dir=<dir>` | `cwd` | Project root where `composer.json` lives |
| `--composer=<bin>` | `composer` | Path or name of the Composer binary |
| `--no-resolve` | off | Skip `composer update` re-resolution for lock conflicts |
| `--type=json\|lock\|vendor` | auto-detect | Override file type detection |
| `-h, --help` | | Print usage |
| `-V, --version` | | Print version |

## How it works

### `composer.json`

Three-way merge at the JSON object level — not line-by-line.

- **`require` / `require-dev` / `conflict` / `replace` / `provide` / `suggest`**: Each package constraint is resolved independently using the standard three-way rule (take theirs if only they changed, take ours if only we changed, conflict if both changed differently).
- **`autoload` / `autoload-dev` / `scripts` / `extra` / `config`**: Recursively deep-merged.
- **`repositories`**: Merged as a set, deduplicated by URL/path/type, respecting removals.
- **Other fields**: Standard three-way (unchanged side yields, conflict if both sides differ).

Conflicts that cannot be resolved automatically are reported to stderr. The file is written with "ours" as the best-effort value and the driver exits with code 1 (git marks the file as conflicted for manual resolution).

### `composer.lock`

1. Package lists (`packages`, `packages-dev`) are indexed by name and merged independently per package.
2. A package updated only on one side → take that side's version.
3. A package updated on both sides to different versions → **conflict**.
4. On conflict (unless `--no-resolve`): the driver writes a best-effort merged lock to a **temporary directory**, runs `composer update <conflicted-packages>` there, and uses the result. The real project directory is never touched during re-resolution.
5. `content-hash` is recomputed from the already-merged `composer.json`.

### `vendor/composer/*`

These files (`installed.json`, `autoload_psr4.php`, etc.) are generated artifacts. The driver accepts "ours" and exits cleanly. **Run `composer install` after the merge** to regenerate them from the merged lock file.

A post-merge git hook is a convenient place for this:

```bash
# .git/hooks/post-merge
#!/bin/sh
composer install --no-scripts 2>/dev/null
```

---

## Exit codes

| Code | Meaning |
|---|---|
| `0` | Clean merge — no conflicts |
| `1` | Unresolved conflicts remain — git treats the file as conflicted |
| `2` | Fatal error (unreadable file, invalid JSON in all three inputs, etc.) |

```bash
docker build -t composer-merge-driver .
```
